# LEXO — базовый план перехода от Windows-интерфейса к iPhone

## 1. Главная идея

Не делать «ветку iOS вслепую» как отдельное приложение.
Делать одну базу проекта, разделённую на слои:

* ядро
* платформенные адаптеры
* интерфейсы

## 2. Правильная модель проекта

### Общая база (единая для всех платформ)

Сюда входит:

* модели данных книги
* состояние чтения
* логика перевода
* логика TTS
* словарь
* загрузка/сохранение книг
* работа с облачным Python/LLM/TTS сервисом
* навигационная логика
* контроллеры и сервисы

### Платформенные адаптеры

Отдельно по платформам:

* Windows
* iPhone/iPad
* Android

Сюда входит:

* выбор файлов
* локальные пути
* открытие папок
* работа с системным аудио
* разрешения
* методы вызова платформенных функций
* интеграция с системными API

### Интерфейсы

Нужно иметь 2 семейства интерфейсов:

* Desktop UI
* Mobile UI

Desktop UI:

* Windows сейчас
* позже при желании macOS

Mobile UI:

* iPhone
* Android

## 3. Что НЕ делать

* не тащить текущий Windows UI на iPhone
* не делать отдельную логику приложения только для iPhone
* не смешивать платформенные вызовы прямо в экранах
* не пытаться сейчас делать всё приложение целиком под iPhone

## 4. Что делать сейчас

### Этап 1. Зафиксировать архитектуру

Нужно мысленно разделить проект на 3 блока:

1. Core
2. Platform
3. UI

### Этап 2. Определить целевые интерфейсы

Нужно принять правило:

* Windows = отдельный desktop-формат
* iPhone/Android = общий mobile-формат

То есть:

* Mobile UI = примерно 90% общий
* Desktop UI = отдельный

### Этап 3. Сделать первый Mobile UI отдельно от Windows UI

Нужен не весь проект, а только первый каркас:

* главный экран
* верхняя зона
* зона текста
* нижняя панель действий
* экран книги
* базовая навигация

### Этап 4. Подключить к нему существующую общую логику

Не переписывать перевод/TTS/состояние заново.
Подключать к новому mobile UI уже существующее ядро.

### Этап 5. Изолировать платформенные вызовы

Сразу вынести всё, что сейчас жёстко привязано к Windows:

* пути
* окна выбора файлов
* запуск локальных процессов
* десктопные обработчики
* особенности клавиатуры/мыши

## 5. Как видеть картинку под iPhone

Нужно представить не iPhone-версию всего продукта, а первый mobile-поток:

1. Открытие приложения
2. Список книг или текущая книга
3. Экран чтения
4. Тап по слову
5. Показ перевода
6. Озвучка
7. Переходы между экранами

То есть сначала строится не «полный продукт», а mobile-сценарий пользователя.

## 6. Базовое разделение экранов

### Desktop

* широкий layout
* панели рядом
* больше постоянных видимых блоков
* мышь и клавиатура

### Mobile

* вертикальный layout
* один главный экран за раз
* нижние панели / bottom sheet
* крупные тач-зоны
* меньше одновременно видимых блоков

## 7. Что должно совпадать между платформами

Общее:

* данные
* логика
* структура книги
* состояние чтения
* перевод
* словарь
* TTS
* API к облачному сервису

Разное:

* layout
* жесты
* размеры блоков
* способ выбора файлов
* системные вызовы
* некоторые методы запуска функций

## 8. Первый реальный результат этапа

Нужно получить не «готовый iOS-продукт», а это:

* отдельный Mobile UI каркас
* подключение к общей логике
* один рабочий mobile-экран чтения
* понимание, какие части Windows-кода надо вынести в адаптеры

## 9. Формула проекта

Одна база приложения
+
Desktop UI
+
Mobile UI
+
Platform adapters

## 10. Следующая мысль для развития

После этого можно будет идти так:

1. собрать mobile UI
2. вычистить Windows-only зависимости
3. подключить iPhone build pipeline
4. сделать первую установку на iPhone
5. потом уже доводить Android
