# MVP2 — план реализации

## Цель этапа

Собрать локальный pipeline двуязычной книги:

1. пользователь выбирает `.txt`
2. backend импортирует текст один раз
3. текст нормализуется и разбивается на абзацы
4. каждый абзац разбивается на предложения
5. предложения переводятся локальной моделью
6. сегменты и абзацы сохраняются в SQLite
7. reader показывает английский и русский абзацы
8. при повторном открытии перевод не запускается повторно

## Что считаем входом и выходом

### Вход

- `title`
- `source_lang`
- `target_lang`
- `text`

### Выход

- запись книги в SQLite
- список абзацев с `source_text` и `target_text`
- список sentence segments для каждого абзаца
- статус обработки книги

## Рабочие допущения на старт

### 1. Режим книги

- UI пока остаётся в режиме одной текущей книги
- storage проектируется так, чтобы не ломаться при переходе к нескольким книгам

### 2. Сегментация предложений

- на первом проходе используем локальный простой сегментатор на regex/эвристиках
- если качество окажется слабым, выносим это в отдельный этап улучшения

### 3. Статусы книги

- `empty`
- `processing`
- `ready`
- `error`

### 4. Translation backend

- сначала делаем интерфейс и mock-провайдер
- потом подключаем `NllbProvider`
- `OpusMarianProvider` идёт вторым backend-слоем после рабочего NLLB-контура

### 5. Модель NLLB

Нужно закрыть отдельным практическим шагом:

- где хранить исходную модель
- куда сохранять CTranslate2-конвертацию
- каким скриптом запускать конвертацию
- как engine выбирает путь к модели

## Этапы реализации

## Этап 0. Контракт и каркас

### Задачи

- зафиксировать API для импорта и чтения книги
- определить DTO для книги, абзаца и сегмента
- решить, оставляем ли текущий `http.server` или переводим backend на `FastAPI`

### Рекомендация

- перейти на `FastAPI`, потому что он уже указан в `MVP2.txt`

### Критерий готовности

- есть согласованный контракт запросов и ответов
- есть понятная точка входа backend

## Этап 1. Storage layer

### Задачи

- переделать схему SQLite под:
  - `books`
  - `paragraphs`
  - `segments`
- добавить хранение `status`, `model_name`, `created_at`
- обеспечить повторное открытие книги без повторного перевода

### Критерий готовности

- backend умеет сохранить книгу, абзацы и сегменты в новой схеме
- reader может получить paragraph-based данные из БД

## Этап 2. Text loading and segmentation

### Задачи

- вынести очистку текста в `text_loader.py`
- вынести разбиение на абзацы и предложения в `segmenter.py`
- убрать эту логику из монолитного `storage.py`

### Критерий готовности

- pipeline получает на вход raw text
- на выходе даёт чистые абзацы и sentence segments

## Этап 3. Translation abstraction

### Задачи

- добавить интерфейс `TranslationProvider`
- реализовать `MockProvider` для локальных быстрых проверок
- подготовить `NllbProvider`
- подготовить задел под `OpusMarianProvider`

### Критерий готовности

- любой провайдер принимает `list[str]` сегментов и возвращает `list[str]` переводов

## Этап 4. Paragraph assembly

### Задачи

- реализовать `assembler.py`
- собирать `paragraph.target_text` из переведённых сегментов
- хранить и сегменты, и итоговые абзацы

### Критерий готовности

- каждый абзац имеет `source_text`
- каждый абзац имеет корректно собранный `target_text`

## Этап 5. Backend API

### Задачи

- обновить endpoint импорта книги
- обновить endpoint чтения абзацев
- убрать из основного сценария старые MVP1-фичи:
  - lookup слова
  - TTS
  - saved words

### Критерий готовности

- UI может импортировать книгу
- UI может открыть bilingual reader

## Этап 6. Flutter UI

### Задачи

- обновить `Home` под новый статус обработки
- обновить `Reader` под показ:
  - исходного абзаца
  - переведённого абзаца
- временно отцепить элементы старого word/TTS-сценария из основного флоу

### Критерий готовности

- пользователь загружает `.txt`
- после обработки видит двуязычные абзацы в reader

## Этап 7. Подключение NLLB

### Задачи

- определить директории модели
- скачать исходную модель
- сконвертировать её в формат CTranslate2
- подключить `AutoTokenizer`
- реализовать language-code mapping
- прогнать импорт и перевод на реальном `.txt`

### Критерий готовности

- перевод реально выполняется локально через NLLB
- сегменты и абзацы сохраняются в SQLite

## Этап 8. Проверка и доводка

### Задачи

- проверить повторное открытие книги без повторного перевода
- проверить пустые строки, мусорные блоки и короткие абзацы
- проверить стабильность API и UI

### Критерий готовности

- сценарий из `MVP2.txt` проходит end-to-end

## Практический блок по NLLB

Это нужно сделать отдельно и по порядку:

1. Подготовить Python-окружение для engine
2. Установить зависимости:
   - `ctranslate2`
   - `transformers`
   - `sentencepiece`
   - `sacremoses`
   - `fastapi`
   - `uvicorn`
3. Выбрать каталог для моделей, например `data/models/`
4. Скачать `facebook/nllb-200-distilled-600M`
5. Сконвертировать модель в CTranslate2-формат
6. Подключить её в `translator.py`
7. Прогнать тестовый перевод 2-3 предложений
8. После этого встраивать в pipeline импорта книги

## Порядок, в котором предлагаю идти

1. Сначала storage + segmentation + mock translation
2. Затем backend API и reader под новую bilingual-схему
3. Затем подключение реальной NLLB
4. Затем практическая доводка импорта и повторного открытия

## Что делаем следующим шагом

Следующий рабочий этап:

- перевести backend на новую схему данных
- вынести text pipeline по отдельным модулям
- подключить временный mock translation provider

После этого уже есть смысл подключать реальную NLLB и заниматься скачиванием, конвертацией и связью с engine.
