# LEXO — история за 2026-03-31

## Сессия

Подготовка плана реализации `MVP2` для перехода от MVP1 к локальному pipeline двуязычной книги

## Что изменено

- прочитан `MVP2.txt` и сопоставлен с текущим состоянием проекта
- зафиксировано, что текущая реализация ещё соответствует старому MVP:
  - абзацы
  - word lookup
  - saved words
  - stub TTS
- создан отдельный план:
  - `MVP2_IMPLEMENTATION_PLAN.md`
- в план вынесены:
  - целевая архитектура
  - этапы реализации
  - критерии готовности
  - практический блок по подключению `NLLB`
- backend переведён на новую storage-схему `MVP2`:
  - `books`
  - `paragraphs`
  - `segments`
- база перенесена на файл:
  - `data/lexo.db`
- добавлены новые модули pipeline:
  - `engine/text_loader.py`
  - `engine/segmenter.py`
  - `engine/assembler.py`
  - `engine/translator.py`
  - `engine/api.py`
- `engine/storage.py` переписан под bilingual pipeline книги
- добавлен `MockProvider` для локального прогона без реальной модели
- reader API теперь отдаёт абзацы в виде:
  - `source_text`
  - `target_text`
- Flutter `Home` и `Reader` перестроены под сценарий двуязычной книги
- из основного UI-сценария убраны:
  - word-card
  - TTS
  - saved words
- исправлен runtime-крайний случай при импорте уже сохранённого `source.txt`
- выполнен smoke-тест импорта книги через новый storage pipeline

## Зачем

- перед реализацией нужен зафиксированный маршрут работ
- `MVP2` заметно меняет модель данных и backend pipeline
- подключение `NLLB` требует отдельного технического этапа:
  - скачивание
  - конвертация в `CTranslate2`
  - интеграция в engine
- сначала нужен рабочий контур без модели, чтобы изолировать ошибки схемы, API и UI

## Ограничения текущей среды

- из текущего WSL-окружения недоступен `cmd.exe`
- в текущем Linux Python нет `pip` и `ensurepip`
- поэтому реальную установку зависимостей и скачивание модели нужно запускать у пользователя через Windows `CMD`
