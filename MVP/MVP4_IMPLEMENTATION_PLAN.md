# MVP4 — план реализации

## Цель этапа

Перейти от модели одной текущей книги к библиотеке книг:

1. при старте приложения открывается библиотека
2. пользователь видит список импортированных книг
3. пользователь может загрузить новую книгу
4. каждая книга хранится как отдельная сущность с собственным `book_id`
5. reader открывает выбранную книгу, а не глобальную `current-book`
6. перевод, TTS, кэш и позиция чтения не конфликтуют между книгами
7. переключение между книгами не ломает playback и состояние приложения

## Что считаем входом и выходом

### Вход

- `.txt` файл новой книги
- выбранная книга из библиотеки
- действия пользователя:
  - открыть книгу
  - импортировать книгу
  - запустить TTS
  - вернуться в библиотеку

### Выход

- список книг в библиотеке
- отдельный `book_id` для каждой книги
- раздельные данные книги, TTS и кэша
- безопасное переключение между книгами без смешивания состояний

## Почему это нужно

Текущая модель `current-book` уже стала узким местом:

- новая книга перезаписывает предыдущую как активную
- TTS и playback привязаны к одному глобальному `book_id`
- при смене книги возможно смешивание состояния:
  - старый playback
  - старый TTS job
  - новый текст
- это уже мешает нормальному UX и дальнейшему развитию

## Рабочие допущения на старт

### 1. Библиотека нужна раньше, чем новые сложные фичи

- прежде чем расширять TTS и reader дальше, нужно стабилизировать модель хранения книг

### 2. Источник истины

- книга существует в БД как отдельная запись
- все связанные сущности привязаны к `book_id`

### 3. Хранение исходников

- у каждой книги должен быть собственный путь к исходному тексту внутри проекта
- нельзя больше использовать один глобальный `data/current_book/source.txt`

### 4. Смена книги

- при открытии другой книги нужно останавливать текущий player
- reader должен грузить состояние строго выбранной книги
- TTS state должен быть изолирован по `book_id`

## Целевая структура данных

```text
LEXO/
  data/
    lexo.db
    books/
      <book_id>/
        source.txt
    tts/
      <book_id>/
        <engine_id>/
          <voice_id>/
            *.wav
```

## Этапы реализации

## Этап 0. Контракт библиотеки

### Задачи

- определить API для:
  - списка книг
  - импорта новой книги
  - открытия книги
  - получения текущей выбранной книги
- определить, где хранится “последняя открытая книга”

### Критерий готовности

- есть согласованный контракт library -> reader

## Этап 1. Storage refactor

### Задачи

- убрать жёсткую зависимость от `CURRENT_BOOK_ID = "current-book"`
- генерировать реальный `book_id` для каждой книги
- хранить исходный текст в `data/books/<book_id>/source.txt`
- перевести storage и queries на параметризованный `book_id`

### Критерий готовности

- импорт двух книг создаёт две независимые записи, а не перезаписывает одну

## Этап 2. Library API

### Задачи

- добавить endpoints:
  - список книг
  - открыть книгу
  - получить активную книгу
- import должен возвращать не “текущую книгу вообще”, а конкретную импортированную книгу

### Критерий готовности

- backend умеет работать с несколькими книгами

## Этап 3. Home -> Library

### Задачи

- стартовый экран больше не про `current-book`
- он становится библиотекой:
  - список книг
  - кнопка `Load TXT`
  - кнопка открытия выбранной книги
- оставить загрузку новой книги как есть по смыслу

### Критерий готовности

- при старте пользователь видит библиотеку, а не один глобальный слот книги

## Этап 4. Reader per book

### Задачи

- reader должен открываться с конкретным `book_id`
- позиция чтения хранится отдельно для каждой книги
- запросы reader должны идти по выбранной книге

### Критерий готовности

- две книги открываются независимо и не мешают друг другу

## Этап 5. TTS isolation

### Задачи

- `tts_jobs` и `tts_segments` полностью изолировать по `book_id`
- файловый кэш TTS хранить в `data/tts/<book_id>/...`
- при смене книги сбрасывать активный playback player
- не допускать, чтобы completion listener старой книги продолжал слать `next`

### Критерий готовности

- TTS одной книги не влияет на TTS другой книги

## Этап 6. Migration and cleanup

### Задачи

- убрать legacy-путь `data/current_book/`
- мигрировать или переинициализировать текущую тестовую книгу в новую схему
- очистить код от assumptions про один book slot

### Критерий готовности

- в проекте больше нет логики “только одна текущая книга”

## Этап 7. Проверка

### Задачи

- импортировать минимум 2 книги
- открыть первую, затем вторую, затем снова первую
- проверить:
  - reader position
  - перевод
  - TTS
  - кэш
  - отсутствие конфликтов state

### Критерий готовности

- сценарий многокнижной библиотеки проходит без залипаний и смешивания состояний

## Что считаем обязательным для MVP4

- библиотека книг на стартовом экране
- импорт новой книги
- отдельный `book_id` на каждую книгу
- изолированный reader state
- изолированный TTS state
- безопасное переключение между книгами

## Что пока можно не делать

- удаление книг из библиотеки
- редактирование метаданных книги
- поиск по библиотеке
- обложки
- теги/коллекции
- сортировки и фильтры сложнее базовых

## Порядок, в котором предлагаю идти

1. Сначала `storage + book_id + files layout`
2. Затем `library API + home screen`
3. Затем `reader per book`
4. Затем `tts isolation + player reset`
5. Затем тест двух и более книг

## Что делаем следующим шагом

Следующий рабочий этап:

- убрать глобальную модель `current-book`
- перевести storage и TTS на реальные `book_id`
- подготовить библиотечный список книг на стартовом экране

После этого уже можно безопасно переключаться между книгами без конфликтов состояния.
