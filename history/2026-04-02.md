# LEXO — история за 2026-04-02

## Сессия

Привязка `word_id` к target span в reader и устранение смещения перевода на `the/a/an`.

## Что изменено

- В backend добавлен новый модуль:
  - `engine/word_alignment.py`
- В storage добавлены сущности и миграции для word-level данных:
  - `source_words`
  - `word_alignments`
- На этапе импорта книга теперь сохраняет:
  - стабильный `source_word_id` для каждого слова
  - `anchor_source_word_id` для служебных артиклей
  - target alignment на уровне сегмента
- Для `the`, `a`, `an` введено правило наследования привязки от ближайшего content-word:
  - сначала вправо
  - если справа нет, то влево
- `GET /reader/paragraphs` расширен:
  - теперь каждый paragraph возвращает список `words`
  - у каждого слова есть готовые поля `translation_left_text`, `translation_focus_text`, `translation_right_text`
- Flutter reader переведён с локального `wordIndex` на реальные word-объекты из payload.
- Удалён старый UI-helper с пропорциональной эвристикой:
  - `app/lib/src/reader/translation_context_builder.dart`
- Для старых книг без сохранённых word rows добавлен runtime fallback, чтобы reader не ломался до переимпорта книги.
- В `reader_screen.dart` добавлен diagnostic console log на тап по слову:
  - пишет `en`, `ru`, `span`, `anchor`, `left`, `right`
  - нужен для ручного сбора проблемных alignment-кейсов из реального UI
  - затем вывод переключён с `developer.log` на `debugPrint`, чтобы строки гарантированно были видны в обычной консоли `flutter run`
- Затем `word_alignment.py` доработан для уменьшения расползания привязки:
  - введены hard anchors для `Chapter`, чисел, `AM/PM` и имён (`Tom`, `Luna`, `Anna`)
  - общее выравнивание заменено на локальные окна между anchor-точками
  - `AM/PM` отделены от copula-слова `am`, чтобы не ломать привязку времени
  - `the/a/an` и copula-формы по-прежнему наследуют ближайший content anchor
- Затем поверх word-level alignment введены `tap units` для выбора смысловых блоков:
  - `pronoun + be` (`he is`)
  - `article + noun` (`the sun`)
  - `time block` (`7 00 AM`)
  - `chapter + number` (`Chapter 1`)
  - whitelist phrasal verbs (`wakes up`, `comes back`, ...)
- Затем логика `be` уточнена по направлению блока:
  - `pronoun + be` остаётся левым блоком (`He is -> Он`)
  - после существительного/имени `be` уходит вправо в предикатный блок (`is bright`, `is small`)
- Затем tap-unit логика уточнена ещё раз:
  - `it is` убрано из специального левого блока `pronoun + be`
  - вместо частного правила добавлен более общий grammar-like noun-phrase scenario:
    - `article + adjective* + noun`
    - блок строится только если следующая граница похожа на boundary, а не на продолжение phrase
  - для unit-перевода `translation_span_text` и верхний `ru` теперь берутся из одного и того же target-range, а не из разных источников
- Reader payload теперь отдаёт для каждого слова:
  - `tap_unit_id`
  - `source_unit_text`
- Flutter selection переведён с `word_id` на `tap_unit_id`:
  - клик по любому слову внутри блока выбирает весь блок
  - diagnostic log теперь показывает и исходное слово, и весь source unit
- Подсветка reader переведена с по-словного `Container` на `TextSpan`-заливку по всему `tap unit`:
  - пробелы внутри выбранного блока теперь входят в highlight
  - двойная заливка с разрывом между словами убрана
- Токенизация слов в backend и Flutter расширена под дефисные формы:
  - `T-Shirt`
  - `well-known`
  - подобные им слова больше не режутся на два кликабельных токена

## Проверка

- `python3 -m compileall engine` проходит.
- Выполнен backend smoke-тест на временной директории:
  - импорт книги проходит
  - `get_paragraphs()` возвращает `words`
  - артикли получают `anchor_word_id`
  - translation context собирается из нового word-level payload
- Выполнен дополнительный unit-like smoke-тест `build_word_mappings()` на типовых кейсах:
  - `Chapter 1 ... Tom ... 7 00 AM`
  - `Good morning Luna says Tom`
  - `At 10 00 AM Tom goes to the park`
  - hard anchors и время теперь удерживаются корректнее
- Выполнен smoke-тест `build_tap_word_payloads()`:
  - `He` и `is` собираются в один tap-unit `He is -> Он`
  - `7`, `00`, `AM` собираются в один tap-unit `7 00 AM -> 7 00 утра`
- Выполнен дополнительный smoke-тест по `be`-блокам:
  - `The sun` / `is bright`
  - `Luna` / `is small`
  - `He is` / `happy`
- Выполнен дополнительный smoke-тест по новым unit-правилам:
  - `It is a beautiful day` -> `a beautiful day`
  - `The Sunny Morning` -> единый блок
  - `T-Shirt` -> один токен, не два
- Затем grammar-like классификация скорректирована:
  - убран слишком широкий adjective-suffix `-y`, из-за которого `day` ошибочно считался adjective-like
  - после этого `a beautiful day` и `The New Day` снова собираются как noun phrases
- Создан новый стратегический документ этапа:

  - два UI-семейства: `Desktop UI` и `Mobile UI`
  - `Android Emulator` как mobile preview на Windows
  - подготовка базы для будущего iPhone pipeline
  - правило синхрона новых функций через `shared state/actions`, а не через отдельные локальные UI-ветки
- Начат первый структурный рефактор по `MVP8` в Flutter-части:
  - добавлены shared feature-контракты:
    - `app/lib/src/features/library/library_feature.dart`
    - `app/lib/src/features/reader/reader_feature.dart`
  - `home_screen.dart` переведён на `LibraryFeatureState/Controller`
  - `reader_screen.dart` переведён на `ReaderFeatureState/Controller`
  - текущий desktop UI сохранён как рабочий, но логика загрузки и основных действий больше не живёт только внутри screen-файлов
- Затем добавлен первый живой `Mobile UI` каркас:
  - `app/lib/src/ui/mobile/screens/mobile_library_screen.dart`
  - `app/lib/src/ui/mobile/screens/mobile_reader_screen.dart`
  - в `app.dart` добавлен adaptive root:
    - узкий экран открывает `MobileLibraryScreen`
    - широкий экран открывает текущий desktop `HomeScreen`
  - mobile library использует тот же shared `LibraryFeatureController`
  - mobile reader использует тот же shared `ReaderFeatureController`
  - TTS в mobile пока подан через bottom sheet с существующим `TtsPanel`
- Затем mobile-подача переработана под более чистый сценарий чтения:
  - добавлен отдельный экран настроек:
    - `app/lib/src/ui/mobile/screens/mobile_settings_screen.dart`
  - из `mobile_reader_screen.dart` убрана постоянная нижняя TTS-панель-кнопка
  - в mobile reader доступ к TTS и reader-настройкам перенесён в `AppBar` через шестерёнку
  - в `mobile_library_screen.dart` добавлена та же точка входа в настройки
  - загрузка `.txt` для mobile вынесена из `FAB` библиотеки в экран настроек
  - в результате основной mobile reader теперь ближе к схеме:
    - книга + текст на основном экране
    - TTS/загрузка/служебные действия в отдельном settings-flow
- Затем собран первый постоянный `mobile shell`:
  - `app/lib/src/ui/mobile/screens/mobile_shell_screen.dart`
  - в `app.dart` mobile root переключён с одиночного `MobileLibraryScreen` на `MobileShellScreen`

    - экран-вкладка внутри shell

  - если книга ещё не выбрана, вкладка `Reader` показывает placeholder вместо пустого reader
  - импорт из mobile settings теперь триггерит reload библиотеки через `reloadTick`, чтобы library-tab мог подтянуть новые книги
- Для Windows-предпросмотра mobile UI добавлены отдельные `.bat`-скрипты:
  - `install_lexo_mobile_android.bat`
  - `fix_lexo_mobile_env.bat`
  - `setup_lexo_mobile_android.bat`
  - `run_lexo_mobile_android.bat`
- `install_lexo_mobile_android.bat` добавлен как внешний install-шаг:
  - пытается поставить `Android Studio` через `winget`
  - затем ищет `Android SDK`
  - ставит Android packages через `sdkmanager`
  - создаёт AVD `LEXO_Pixel_6`
  - после этого вызывает `setup_lexo_mobile_android.bat`
- `fix_lexo_mobile_env.bat` добавлен для постоянной настройки Windows-среды:
  - сохраняет `JAVA_HOME` через `setx`
  - добавляет в user `Path`:
    - `%JAVA_HOME%\bin`
    - `C:\src\flutter\bin`
  - нужен, чтобы `run_lexo_mobile_android.bat` запускался повторно без ручных `set JAVA_HOME` в каждом новом `CMD`
- `setup_lexo_mobile_android.bat` делает следующее:
  - теперь позиционируется как project-setup для уже установленной Android-среды
  - ищет `Flutter SDK`
  - ищет `Android SDK`
  - включает Android support через `flutter config --enable-android`
  - при отсутствии `app/android` создаёт Android-платформу через `flutter create --platforms=android .`
  - запускает `flutter doctor`
  - пытается установить Android packages через `sdkmanager`
  - создаёт AVD `LEXO_Pixel_6`
- `run_lexo_mobile_android.bat` делает следующее:
  - находит Python / Flutter / Android SDK
  - проверяет наличие `app/android`
  - поднимает Android Emulator `LEXO_Pixel_6`
  - поднимает `Python engine`
  - запускает `flutter run` уже на Android device, а не на Windows desktop
- Затем mobile `.bat`-скрипты усилены автоподхватом Java:
  - `install`, `setup` и `run` теперь сначала пытаются взять `JAVA_HOME` из уже заданной среды
  - если его нет, они автоматически используют встроенный JBR Android Studio:
    - `C:\Program Files\Android\Android Studio\jbr`
  - затем временно добавляют `%JAVA_HOME%\bin` в `PATH` текущего процесса
  - это снимает зависимость от ручного `set JAVA_HOME=...` для разовых запусков
- Затем `run_lexo_mobile_android.bat` усилен для диагностики падений:
  - на первом уровне он теперь сам перезапускает себя с выводом в файл:
    - `logs/run_lexo_mobile_android.log`
  - после завершения показывает `exit code`
  - при неуспехе не закрывается сразу, а оставляет `pause` и путь к лог-файлу
- Затем исправлен ещё один batch-bug в logging-обёртке `run_lexo_mobile_android.bat`:
  - путь к лог-файлу и `exit code` внутри условного блока читались через `%VAR%`
  - из-за parse-time expansion переменные оказывались пустыми
  - чтение переведено на `!VAR!`, чтобы логирование реально работало
- Затем logging-обёртка `run_lexo_mobile_android.bat` упрощена ещё раз:
  - позднее рекурсивная схема логирования вообще убрана как слишком хрупкая для Windows `cmd`
  - запуск разделён на два файла:
    - `run_lexo_mobile_android.bat` — внешний launcher с логированием
    - `run_lexo_mobile_android_inner.bat` — реальный запуск emulator/engine/flutter
  - это устраняет ошибку `The system cannot find the batch label specified - resolve_python`
- Затем `run_lexo_mobile_android_inner.bat` упрощён ещё раз:
  - убраны все `call :subroutine`-метки
  - базовые проверки Python / Flutter / Java / Android SDK развернуты в линейный batch-flow
  - это устраняет последний класс ошибок Windows `cmd`, связанных с batch-label resolution
- Затем исправлен mobile API endpoint для Android Emulator:
  - `app/lib/src/api/api_client.dart` больше не использует один жёсткий `127.0.0.1`
  - default base URL теперь выбирается по платформе:
    - `Android` -> `http://10.0.2.2:8765`
    - остальные платформы -> `http://127.0.0.1:8765`
  - это позволяет mobile emulator видеть тот же Python engine на Windows-host, что и desktop UI
- Затем Windows `.bat`-скрипты для mobile-предпросмотра разложены по папкам:
  - в корне проекта оставлены только launcher-файлы:
    - `run_lexo_mvp.bat`
    - `run_lexo_mobile_android.bat`
  - вспомогательные mobile-скрипты перенесены в `scripts/`:
    - `scripts/install_lexo_mobile_android.bat`
    - `scripts/setup_lexo_mobile_android.bat`
    - `scripts/fix_lexo_mobile_env.bat`
    - `scripts/run_lexo_mobile_android_inner.bat`
  - пути внутри них скорректированы под новый `project root`, чтобы запуск не зависел от текущей папки скрипта
- Затем исправлен runtime-bug в `setup_lexo_mobile_android.bat`:
  - после `flutter create --platforms=android .` batch-скрипт неверно читал `%ERRORLEVEL%` внутри блока и останавливался, хотя `app/android` уже создавался успешно
  - чтение кода возврата переведено на `!ERRORLEVEL!`
  - добавлен дополнительный guard: если `app/android` уже появился, setup не падает даже при некорректном промежуточном коде возврата
- `dart` и `flutter` в текущем окружении отсутствуют, поэтому автоматическая проверка Flutter-кода не запускалась.

## Ограничения текущей версии

- Текущий alignment в MVP всё ещё эвристический, не полноценный linguistic word-alignment engine.
- Основной фикс сделан правильно по архитектуре:
  - `source_word_id` стал явной сущностью
  - UI больше не вычисляет перевод по относительной позиции слова
- Но точность target-span для сложных перестроений фразы всё ещё ограничена качеством текущей эвристики.
