# MVP3 — план реализации

## Цель этапа

Собрать локальный TTS pipeline для уже импортированной книги:

1. пользователь открывает книгу
2. reader показывает текст книги
3. пользователь нажимает `Play`
4. backend разбивает текст на TTS-сегменты
5. локальный TTS-движок генерирует аудио по сегментам
6. аудио сохраняется внутри `LEXO/data/tts/`
7. reader воспроизводит сегменты по порядку
8. при повторном запуске готовые аудиофайлы используются повторно
9. пользователь может выбрать голос, поставить паузу, продолжить и перейти к следующему или предыдущему сегменту

## Что считаем входом и выходом

### Вход

- `book_id`
- `engine_id`
- `voice_id`
- `mode`
- текущая позиция чтения

### Выход

- TTS job в SQLite
- список TTS-сегментов и их статусов
- аудиофайлы на диске
- управление воспроизведением из reader

## Рабочие допущения на старт

### 1. Базовый TTS backend

- первый рабочий backend: `Kokoro`
- архитектура строится так, чтобы потом можно было подключить другой TTS provider

### 2. Режимы на первом проходе

- обязательно: `play_from_current`
- желательно: `generate_book`
- `generate_chapter` можно держать как задел, если в текущей модели данных ещё нет главы как сущности

### 3. Сегментация для озвучки

- TTS-сегменты не обязаны совпадать с translation-segments из `MVP2`
- TTS-сегменты строятся отдельно под естественную речь и ограничения движка

### 4. Кэширование

- аудио кэшируется файлами, не blob-полями в SQLite
- повторная генерация не должна запускаться, если есть готовый файл для того же текста, движка и голоса

### 5. UI-первый сценарий

- reader сначала получает стабильный playback-контур
- сложные фоновые режимы генерации вторичны относительно `play/pause/resume/next/prev`

## Этапы реализации

## Этап 0. Контракт и каркас

### Задачи

- зафиксировать TTS API для:
  - списка голосов
  - запуска озвучки
  - получения очереди сегментов
  - статуса job
- определить DTO для:
  - `tts_profile`
  - `tts_job`
  - `tts_segment`
  - playback state

### Критерий готовности

- есть согласованный API-контракт между engine и reader
- есть понятная схема того, как reader управляет playback

## Этап 1. Data layer

### Задачи

- добавить в SQLite таблицы:
  - `tts_profiles`
  - `tts_jobs`
  - `tts_segments`
- создать файловую структуру для аудио в `data/tts/`
- продумать ключ кэша:
  - `book_id`
  - `segment hash`
  - `engine_id`
  - `voice_id`

### Критерий готовности

- backend умеет хранить TTS jobs и сегменты
- аудиофайлы сохраняются в проектную папку

## Этап 2. TTS segmentation

### Задачи

- реализовать `tts_segmenter.py`
- использовать политику:
  - `target_chunk_chars = 160`
  - `soft_max_chunk_chars = 220`
  - `hard_max_chunk_chars = 320`
- сначала учитывать абзацы
- затем предложения
- затем аккуратно дорезать длинные куски по безопасным знакам препинания

### Критерий готовности

- длинный текст разбивается на естественные сегменты
- сегменты пригодны для качественной озвучки

## Этап 3. Provider abstraction

### Задачи

- добавить `tts_provider.py` с единым интерфейсом
- реализовать `MockTtsProvider` для быстрого прогона без реальной модели
- подготовить `kokoro_provider.py`

### Критерий готовности

- любой TTS provider принимает текст и голос
- на выходе даёт путь к аудио и служебные данные

## Этап 4. Cache and queue

### Задачи

- реализовать `tts_cache.py`
- реализовать `tts_queue.py`
- научить queue manager:
  - пропускать готовые сегменты
  - ставить в генерацию только отсутствующие
  - сохранять статусы `pending / processing / ready / error`

### Критерий готовности

- повторный запуск озвучки не генерирует уже готовые сегменты
- очередь управляет генерацией предсказуемо

## Этап 5. Backend API

### Задачи

- добавить endpoints для:
  - списка голосов
  - запуска TTS playback pipeline
  - получения готовых сегментов
  - переключения текущего сегмента
- встроить TTS слой в существующий engine

### Критерий готовности

- UI может запросить озвучку и получить сегменты с путями к файлам

## Этап 6. Flutter playback

### Задачи

- добавить выбор голоса в reader
- добавить кнопки:
  - `Play`
  - `Pause`
  - `Resume`
  - `Next`
  - `Prev`
- реализовать последовательное проигрывание сегментов
- отображать текущий статус озвучки

### Критерий готовности

- пользователь управляет озвучкой из reader без знания деталей backend

## Этап 7. Kokoro setup

### Задачи

- определить способ локальной установки `Kokoro` внутри `LEXO`
- хранить модель и runtime внутри проекта
- подготовить Windows `CMD`-скрипт установки
- связать project `.venv` и TTS backend так же, как это уже сделано для `NLLB`

### Критерий готовности

- `Kokoro` устанавливается и запускается локально внутри проекта
- engine может генерировать реальное аудио

## Этап 8. Проверка и доводка

### Задачи

- проверить playback на реальной книге
- проверить повторное использование кэша
- проверить переключение голосов
- проверить `pause/resume/next/prev`
- проверить поведение после перезапуска приложения

### Критерий готовности

- сценарий из `MVP3.txt` проходит end-to-end

## Порядок, в котором предлагаю идти

1. Сначала `storage + tts tables + tts segmenter + mock TTS`
2. Затем `backend API + reader playback controls`
3. Затем `Kokoro setup` и реальная генерация аудио
4. Затем кэш, доводка и проверка повторного использования

## Что делаем следующим шагом

Следующий рабочий этап:

- добавить TTS data layer
- вынести TTS pipeline в `engine/tts/`
- подключить временный `MockTtsProvider`
- подготовить reader к playback-контролам

После этого уже есть смысл подключать реальный `Kokoro` и заниматься его установкой, кэшем и локальным запуском внутри `LEXO`.
