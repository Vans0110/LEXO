# LEXO — история за 2026-04-03

## Сессия

Формализация `MVP9` для mobile book package, локального mobile storage и desktop-host processing model.

## Что изменено

- проанализировано текущее состояние mobile-ветки относительно iPhone/mobile runtime-модели
- зафиксировано, что текущая mobile-схема ещё зависит от desktop host через:
  - `source_path` при импорте книг
  - `audio_path` при mobile TTS playback
  - `127.0.0.1` как fallback base URL вне Android Emulator
- согласована новая продуктовая модель:
  - mobile хранит книги локально
  - desktop используется как processing-host
  - обработанная книга возвращается на mobile как переносимый package
  - desktop library может экспортироваться на mobile как локальная копия
- создан новый стратегический документ этапа:
  - `MVP/MVP9_MOBILE_BOOK_PACKAGE_PLAN.md`
- в `MVP9` зафиксировано:
  - формат `book package`
  - сценарий `Import from Mobile File`
  - сценарий `Import from Desktop Library`
  - новый API contract для mobile import/export/package download
  - отказ от desktop file paths как runtime-контракта mobile
  - переход mobile reader на локальный package как источник истины
  - новый TTS contract через local audio или explicit audio download
  - критерии готовности этапа и порядок реализации
- начата практическая реализация `MVP9`
- в backend добавлены новые mobile endpoints:
  - `POST /mobile/books/import-text`
  - `GET /mobile/desktop-books`
  - `GET /mobile/books/package`
  - `GET /mobile/books/audio`
- `engine/storage.py` расширен под:
  - импорт книги из переданного текста без `source_path`
  - сборку `book package` для mobile
  - выдачу пути к TTS audio segment для отдельного download endpoint
  - mobile `tts_manifest` snapshot внутри package
- в Flutter добавлен локальный mobile storage:
  - `app/lib/src/mobile/mobile_package_models.dart`
  - `app/lib/src/mobile/mobile_package_repository.dart`
- mobile import из `Settings` больше не отправляет на backend `source_path`
  - mobile читает `.txt` локально
  - отправляет на host текст
  - получает `book package`
  - сохраняет его в локальную mobile library
- добавлен второй mobile import-сценарий:
  - импорт готовой книги из desktop library через package download
- `MobileLibraryScreen` переведён с host-библиотеки на локальную mobile library
- `MobileReaderScreen` переведён на чтение локального package:
  - reader payload больше не загружается из desktop API
  - позиция чтения сохраняется в локальный package
  - при наличии desktop origin дополнительно пробрасывается в host для TTS `play_from_current`
- mobile TTS playback больше не открывает host `audio_path` напрямую
  - mobile скачивает нужный audio segment с host endpoint
  - сохраняет `.wav` локально
  - воспроизводит уже локальный файл
- добавлены локальные mobile app settings:
  - `app/lib/src/mobile/mobile_settings_repository.dart`
  - сохранение пользовательского `Host URL` для подключения к desktop host
- `LexoApiClient` переведён на настраиваемый runtime `baseUrl`, чтобы mobile можно было переключать на IP Windows-host для реального iPhone
- в `MobileSettingsScreen` добавлен отдельный блок host/sync:
  - просмотр текущего `Host URL`
  - ручное редактирование адреса
  - ручное обновление текущей mobile-книги из desktop library
- в mobile library добавлен ручной update flow по `desktop_book_id`
  - synced-книги помечаются отдельно
  - книгу можно обновить без повторного импорта
  - при update сохраняется локальная reading position
- подготовлен базовый LAN-режим для реальной iPhone-интеграции
- `engine/main.py` теперь поддерживает runtime bind через env:
  - `LEXO_HOST`
  - `LEXO_PORT`
- `scripts/run_lexo_mobile_android_inner.bat` теперь поднимает engine с bind на `0.0.0.0:8765`, а не только на loopback
- добавлен отдельный launcher:
  - `run_lexo_engine_lan.bat`
  - нужен для запуска только Python engine в LAN-режиме без Android Emulator
- добавлен helper script для подготовки чистой GitHub-копии проекта:
  - `make_github_repo.bat`
  - собирает рядом папку `LEXO_GITHUB` без локальных env, data, logs, build-artifacts и ephemeral-файлов
- desktop Flutter flow сохранён отдельно и не переведён на mobile package model

## Ограничения текущей реализации

- `flutter analyze` и `dart format` не запускались:
  - в текущем окружении отсутствуют `flutter` и `dart`
- сам runtime `Host URL` уже настраиваемый, но реальная iPhone-интеграция ещё требует проверки сети:
  - IP Windows-host
  - доступность порта `8765`
  - разрешение фаервола
- фактическая проверка с real iPhone ещё не выполнялась из текущего окружения
- mobile reader уже локальный по книге, но TTS generation/control всё ещё идут через desktop host как processing service
- desktop library sync сейчас ручной:
  - import package по запросу
  - без автообновления и без diff-синхронизации
- GitHub Actions workflow пока собирает unsigned iOS artifact:
  - для реальной установки на iPhone ещё нужен шаг ручной подписи через `Sideloadly`
- текущая iPhone-проверка ещё не завершена:
  - после сетевой iOS-правки нужен повторный прогон на устройстве

## Зачем

- текущая схема подходит для desktop и частично для Android Emulator preview, но не подходит как основа для реального iPhone runtime
- mobile-приложение не должно зависеть от файловых путей desktop-хоста
- нужен переносимый формат книги, чтобы tap units, translation context и TTS могли жить локально на устройстве
- нужен отдельный этап перед iPhone build pipeline, чтобы не тащить в iOS неправильные desktop-зависимости

## Ожидаемый результат

- следующий рабочий этап реализации идёт по утверждённой модели `desktop processing -> mobile package -> local mobile reader`
- mobile import и mobile TTS будут перепроектированы без зависимости от desktop filesystem
- появится база для локальной mobile library и синка книг из desktop-библиотеки
- после этого проект можно будет переводить к реальному iPhone pipeline без архитектурного долга по file paths
