• 1. Общая архитектура

  Дерево проекта, верхний уровень:

  - engine — Python backend: импорт книги, перевод, mapping, storage, API, TTS.
  - app — Flutter UI для desktop/mobile shell, reader, cards.
  - MVP — продуктовые и технические планы этапов.
  - history — журнал изменений по дням.
  - Books — входные .txt книги.
  - scripts — bat/cmd-скрипты запуска и setup.
  - ISO%20pack — iOS/IPA-артефакты.
  - logs — runtime-логи.
  - Other — внешние артефакты, не часть исходного LEXO.

  - NLLB включается только если translator_mode() == "nllb" — engine/translator.py:109-113.

  Где вызывается:

  - engine/storage.py:1296-1299 → _build_paragraph_payloads().

  Единица перевода:

  - Перевод идёт по segments — engine/translator.py:22-27.
  - segments в текущем pipeline = предложения после split_sentences() — engine/storage.py:1296-1299, engine/
    segmenter.py:14-20.

  Контекст:

  - В вызов translate_segments() передаётся только список сегментов текущего абзаца — engine/storage.py:1296-1299.
  - Явная передача контекста соседних абзацев/предложений в translator: не найдено.

  Кеширование:

  - Кеш перевода в translator.py: не найдено.
  - Отдельный translation cache в storage.py: не найдено.

  Ограничения, видимые из кода:

  - По умолчанию backend работает не на NLLB, а на MockProvider, если env не переключён — engine/config.py:17-22,
    engine/translator.py:109-113.
  - NLLB поддерживает только en и ru — engine/translator.py:12-15, 116-123.
  - Если translator вернёт число сегментов, не равное входу, импорт книги падает с ошибкой — engine/storage.py:1297-
    1299.
  - Split предложений основан на одном regex и смотрит на заглавную букву/цифру после .!? — engine/segmenter.py:6, 14-
    20. Более сложная сегментация: не найдена.

  4. Система word mapping

  Где строится:

  - engine/word_alignment.py:190-268 → build_word_mappings().

  Алгоритм, фактически:

  - Оба текста токенизируются regex WORD_PATTERN — engine/word_alignment.py:6, 180-187, 195-196.
  - Ищутся hard anchors — engine/word_alignment.py:475-525.
  - Hard anchors сейчас ограничены:
      - chapter, am, pm
      - цифры
      - tom, luna, anna
        — engine/word_alignment.py:496-514.
  - Для anchor слов ищутся target-кандидаты по списку ANCHOR_TRANSLATIONS и по цифровому совпадению — engine/
    word_alignment.py:170-177, 506-525.
  - Между anchor-парами target индексы для content-слов распределяются пропорционально по позиции — engine/
    word_alignment.py:338-345, 389-472.
  - Function words на этапе mapping — только:
      - артикли the/a/an
      - связка is/am/are/was/were
        — engine/word_alignment.py:551-558.
  - Для function words target берётся от ближайшего content anchor; у артикля есть отдельная попытка взять предыдущий
    target function token — engine/word_alignment.py:348-369, 372-386.
  - На выходе для каждого source token создаётся:
      - запись source_words
      - одна alignment-запись с target_start_index, target_end_index, target_text, confidence
        — engine/word_alignment.py:227-267.

  Структура данных:

  - source_words — source token-level rows — engine/storage.py:100-115, 1244-1273.
  - word_alignments — одна row на source_word_id — engine/storage.py:116-123, 1274-1291.
  - Reader word payload содержит:
      - tap_unit_id
      - source_unit_text
      - translation_left_text
      - translation_focus_text
      - translation_right_text
        — engine/word_alignment.py:307-333.

  Как определяется соответствие слов:

  - Через sequential anchors + proportional distribution within windows — engine/word_alignment.py:389-472, 475-525.
  - Не найдено использование attention, embeddings или bilingual alignment model.

  Tap-unit grouping:

  - Строится отдельно от base word mapping — engine/word_alignment.py:561-578.
  - Поддержанные unit patterns:
      - time
      - chapter
      - pronoun_be
      - article_noun_phrase
      - article_noun
      - be_predicate
      - phrasal_verb
        — engine/word_alignment.py:565-573, 581-688.

  Проблемы/ограничения из кода:

  - В word_alignments одна строка на source word, many-to-many mapping не представлена — engine/storage.py:116-123.
  - Anchor-словарь ограничен фиксированным списком — engine/word_alignment.py:170-177, 496-514.
  - Распределение content-слов по target окнам позиционное, а не семантическое — engine/word_alignment.py:338-345, 467-
    472.
  - Function-word inheritance покрывает только articles/copula — engine/word_alignment.py:551-558.
  - Context window для UI ограничен тремя токенами слева и справа — engine/word_alignment.py:282-287.

  - saved_cards — cards для review — engine/storage.py:133-155.
  - tts_profiles, tts_jobs, tts_segments — TTS subsystem — engine/storage.py:156-190.

  - На уровне paragraph — paragraphs.target_text — engine/storage.py:82-89.
  - На уровне segment — segments.target_text — engine/storage.py:90-99.

  10. TTS система

  Где находится:

  - Backend TTS orchestration — engine/tts/tts_service.py:16-476.
  - Provider selection — engine/tts/tts_provider.py:13-136.
  - Segmentation — engine/tts/tts_segmenter.py:48-266.
  - Flutter playback controls — app/lib/src/screens/reader_screen.dart:37-355.

  Какая система используется:

  - По умолчанию mock — engine/config.py:18, 25-26; engine/tts/tts_provider.py:133-136.
  - Реальный provider: kokoro, если LEXO_TTS_PROVIDER == "kokoro" и есть runtime/runner — engine/tts/tts_provider.py:61-
    136.

  Как режется текст:

  - generate_jobs() читает paragraphs книги — engine/tts/tts_service.py:81-108, 468-474.
  - Далее build_tts_chunks():
      - фильтрует видимые paragraphs
      - разбивает paragraph на lines
      - heading lines force-break
      - обычные lines -> split_sentences()
      - sentence -> _split_phrases() по , ; :
      - длинные фразы -> _split_long_phrase()
        — engine/tts/tts_segmenter.py:48-185, 188-266.

  Как генерируется аудио:

  - Для каждого chunk вызывается generate_tts_segment() — engine/tts/tts_service.py:374-388.
  - generate_tts_segment() сначала проверяет cache path, иначе вызывает provider.synthesize(...) — engine/tts/
    tts_queue.py:18-35.
  - Для Kokoro вызывается отдельный runner script с --speed — engine/tts/tts_provider.py:84-121.

  Скорость:

  - Generation rate задаётся в SpeechProfile.rate — engine/tts/speech_profiles.py:6-17, 24-30.
  - Этот rate передаётся в provider.synthesize(..., rate=profile.rate) — engine/tts/tts_queue.py:29-34.
  - На frontend playback speed отдельно меняется через media_kit Player.setRate() — app/lib/src/screens/
    reader_screen.dart:37-52.

  Time-stretch или генерация под скорость:

  - Генерация под скорость есть: rate передаётся в TTS provider — engine/tts/tts_queue.py:29-34, engine/tts/
    tts_provider.py:89-104.
  - Playback time-stretch тоже есть на frontend: _audioPlayer.setRate(...) — app/lib/src/screens/reader_screen.dart:50-
    52, 229, 263-264, 322.
  - Отдельный DSP time-stretch backend-layer: не найдено.

  11. Узкие места и проблемы

  Подтверждённые кодом:

  - По умолчанию переводчик — mock, не NLLB — engine/config.py:17-22, engine/translator.py:109-113.
  - По умолчанию TTS provider — mock, не Kokoro — engine/config.py:18, 25-26, engine/tts/tts_provider.py:133-136.
  - split_sentences() основан на одном regex и не использует полноценный parser — engine/segmenter.py:6, 14-20.
  - Translation cache не найден — engine/translator.py:1-126, engine/storage.py:1293-1353.
  - Mapping хранит только одну alignment row на source word — engine/storage.py:116-123.
  - Mapping-anchors ограничены фиксированным словарём — engine/word_alignment.py:170-177, 496-514.
  - _assign_content_targets() распределяет индексы пропорционально, без отдельной семантической модели — engine/
    word_alignment.py:338-345.
  - TTS level_ids приходят в API и service, но в generate_jobs() фактически используется только profiles =
    [build_profile()] — engine/api.py:142-149, engine/tts/tts_service.py:64-84.
  - word/audio endpoint — stub, не реальная генерация — engine/api.py:187-189.
  - Верхняя translation bar принудительно maxLines: 1, длинный контекст режется многоточием — app/lib/src/widgets/
    translation_context_bar.dart:33-35.
  - Frontend имеет fallback DetailSheetPayload.fromSelection() при ошибке backend detail-sheet — app/lib/src/screens/
    reader_screen.dart:391-397. Это означает, что detail logic существует в двух местах: backend и fallback frontend.
