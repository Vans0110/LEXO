1. Цель этапа

На выходе этого этапа приложение должно работать так:

Пользователь открывает уже импортированную книгу
Reader показывает текст книги
Пользователь нажимает Play
Приложение локально генерирует озвучку текста сегментами
Озвучка воспроизводится последовательно внутри приложения
Сгенерированные аудиофайлы сохраняются внутри проекта LEXO
При повторном запуске книги или главы уже готовая озвучка используется повторно
Пользователь может выбрать голос
Пользователь может поставить паузу, продолжить и перейти к следующему или предыдущему сегменту

Это не словарь, не карточки слов и не облачный TTS. Это именно локальный pipeline качественной озвучки книги.

2. Основная идея этапа

MVP3 должен не просто “подключить TTS”, а ввести отдельный озвучивающий слой приложения.

Правильная логика такая:

книга уже есть в приложении
текст книги сегментируется для TTS
каждый сегмент передаётся локальному TTS-движку
движок возвращает аудио
аудио сохраняется в кэш проекта
reader воспроизводит сегменты в правильном порядке

Ключевая цель этого этапа:
получить локальную, приятную, связную озвучку текста, а не роботизированное чтение по словам.

3. Что использовать

3.1. Базовый TTS-движок для MVP3

Рекомендованный первый вариант:

Kokoro

Почему именно он:

хорошее качество для локального запуска
ориентация на естественную речь
не слишком тяжёлый по сравнению с более крупными TTS-стеками
есть несколько голосов
подходит для английского художественного текста
может использоваться как основной первый провайдер озвучки

3.2. Что не фиксировать жёстко

Не нужно намертво прибивать реализацию только к одному варианту запуска модели.
Агент должен сам выбрать наиболее гибкий и устойчивый способ интеграции Kokoro в LEXO.

Допустимые варианты:
локальный Python service внутри проекта
локальный sidecar process внутри проекта
обвязка через отдельный engine-модуль внутри проекта
иной локальный вариант, если он не ломает архитектуру LEXO

Жёсткое условие только одно:
всё должно жить внутри папки LEXO, со своими скриптами подготовки, установки и запуска.

3.3. Резерв на будущее

Архитектура должна позволять позже подключить:
другие голоса
другой локальный TTS backend
другой способ инференса
другую модель озвучки

Но на этапе MVP3 реализуется один рабочий локальный TTS-контур с Kokoro.

4. Физическая структура решения

Целевая структура должна остаться в той же логике, что и MVP2:

LEXO/
  app/
  engine/
    api.py
    tts/
      tts_provider.py
      kokoro_provider.py
      tts_segmenter.py
      tts_cache.py
      tts_queue.py
      tts_models.py
  data/
    lexo.db
    tts/
  scripts/
    setup_tts.cmd
    run_lexo_mvp.bat

Смысл разделения такой:

app/ ничего не знает о деталях TTS-модели
engine/ знает только TTS pipeline и API
tts_segmenter.py отвечает только за сегментацию текста для озвучки
tts_provider.py задаёт единый интерфейс TTS
kokoro_provider.py реализует конкретную работу с Kokoro
tts_cache.py хранит логику повторного использования аудио
tts_queue.py управляет очередью генерации и воспроизведения
scripts/ отвечают за установку и запуск внутри проекта

5. Что должно появиться в данных

SQLite и файловое хранилище должны хранить не только книгу и перевод, но и TTS-структуру.

5.1. Таблица tts_profiles

Назначение:
хранить список доступных голосов и движков для приложения

Минимальные поля:
id
engine_id
voice_id
display_name
lang
is_enabled

5.2. Таблица tts_jobs

Назначение:
фиксировать факт подготовки озвучки для книги

Минимальные поля:
id
book_id
engine_id
voice_id
status
created_at
updated_at

Пример статусов:
pending
processing
ready
error

5.3. Таблица tts_segments

Назначение:
хранить связь между текстовым сегментом и аудиофайлом

Минимальные поля:
id
book_id
segment_id
engine_id
voice_id
source_text
audio_path
duration_ms
status
hash
created_at

5.4. Почему нужен файловый кэш

Аудио не должно храниться внутри SQLite blob-полями на этом этапе.
Правильнее хранить файлы на диске внутри LEXO/data/tts/, а в базе хранить путь, статус, длительность и хэш.

6. Контракт обработки озвучки

6.1. Вход

В backend поступает запрос примерно такого вида:

{
  "book_id": "book_001",
  "engine_id": "kokoro",
  "voice_id": "af_heart",
  "mode": "play_from_current"
}

Допустимые режимы:
play_from_current
generate_chapter
generate_book

На этапе MVP3 достаточно, если реально будет стабильно работать:
play_from_current
и базовый режим generate_book

6.2. Этапы обработки

Этап A. Получение текста книги

Engine читает уже сохранённую книгу из базы.
Новый импорт книги в этом этапе не реализуется заново, а использует результаты MVP2.

Этап B. TTS-сегментация

tts_segmenter.py

Задача:
разбить текст на пригодные для озвучки фрагменты

Правила:
нельзя читать книгу одним огромным куском
нельзя дробить текст до уровня отдельных слов
нужно сохранять естественные границы речи

Базовая логика:
сначала учитывать абзацы
потом предложения
потом, если сегмент слишком длинный, дорезать его по безопасным знакам препинания

Практическая политика под Kokoro:

target_chunk_chars = 160
soft_max_chunk_chars = 220
hard_max_chunk_chars = 320

Это не абсолютный закон для всех будущих движков, а стартовая политика именно для первого рабочего контура.

Этап C. Подготовка очереди генерации

tts_queue.py

Queue manager должен:
получить список нужных сегментов
понять, какие из них уже есть в кэше
пропустить готовые
отправить отсутствующие сегменты в TTS provider

Этап D. Генерация аудио

kokoro_provider.py

Провайдер должен:
получить текст сегмента
получить voice_id
запустить локальную генерацию
сохранить аудио в файл
вернуть путь к аудио и служебные данные

Этап E. Кэширование

tts_cache.py

Если сегмент уже был озвучен тем же движком и тем же голосом, повторная генерация не должна запускаться.

Ключ кэша должен учитывать минимум:
book_id
segment_id или текстовый hash
engine_id
voice_id

Этап F. Воспроизведение

Reader или audio controller внутри app должен:
взять путь к аудио
проигрывать сегменты по порядку
перейти к следующему сегменту
уметь остановить и продолжить

7. TTS provider abstraction

Нужно ввести единый интерфейс.

Пример логики:

class TtsProvider:
    def list_voices(self) -> list[dict]:
        ...
    def is_ready(self) -> bool:
        ...
    def synthesize(self, text: str, voice_id: str, output_path: str) -> dict:
        ...

Требование этапа:
весь app и queue-контур работают через TtsProvider, а не напрямую через kokoro-specific код.

Это нужно для того, чтобы потом можно было подключить:
Piper
XTTS
другой локальный backend
или иной вариант Kokoro-runtime

8. Что жёстко не фиксировать агенту

Не надо жёстко заставлять агента использовать один конкретный Python пакет, если есть более практичный локальный вариант.

Агент сам должен выбрать:
как именно ставить Kokoro
какой runtime удобнее
какой способ вызова стабильнее для Windows и проекта LEXO
какой формат аудио удобнее для хранения и воспроизведения

Но обязательные рамки такие:
всё должно быть локально
всё должно ставиться проектными скриптами
всё должно жить внутри LEXO
запуск не должен зависеть от ручной магии вне проекта

9. API backend

9.1. GET /tts/voices

Возвращает список голосов для текущего движка

Пример ответа:
{
  "engine_id": "kokoro",
  "voices": [
    {
      "voice_id": "af_heart",
      "display_name": "Female 1",
      "lang": "en"
    }
  ]
}

9.2. POST /tts/generate

Вход:
{
  "book_id": "book_001",
  "engine_id": "kokoro",
  "voice_id": "af_heart",
  "mode": "generate_book"
}

Выход:
{
  "job_id": "tts_job_001",
  "status": "processing"
}

9.3. GET /tts/jobs/{job_id}

Возвращает статус генерации озвучки

Пример:
{
  "job_id": "tts_job_001",
  "status": "ready",
  "generated_segments": 120,
  "total_segments": 120
}

9.4. GET /books/{book_id}/tts-segments

Возвращает reader-ready данные для воспроизведения

Пример:
{
  "book_id": "book_001",
  "engine_id": "kokoro",
  "voice_id": "af_heart",
  "segments": [
    {
      "segment_id": "seg_001",
      "text": "Hello world.",
      "audio_path": "data/tts/book_001/kokoro/af_heart/0001.wav",
      "status": "ready"
    }
  ]
}

10. UI задачи

10.1. Reader screen

Добавить в Reader:
кнопку Play
кнопку Pause
кнопку Next
кнопку Previous
переключатель голоса
индикатор состояния генерации

На этапе MVP3 не нужно делать сложный аудиоплеер.
Достаточно рабочего базового контура управления.

10.2. Поведение Reader

Если озвучка ещё не готова:
показать status generating
или начать проигрывание по мере готовности сегментов

Если сегменты уже готовы:
сразу использовать кэш

10.3. Что не тащить в этот этап

Не делать пока:
синхронизацию слова с аудио
караоке-подсветку
выделение текущего слова
сложный аудиоредактор
фоновые облачные задачи
словарь слов на базе TTS

11. Установка и подготовка внутри проекта

Как и в MVP2, нужен отдельный сценарий подготовки.

Нужен проектный setup-скрипт, например:

scripts/setup_tts.cmd

Он должен:
создать или использовать проектный .venv внутри LEXO
установить зависимости TTS
скачать всё необходимое внутрь проекта
подготовить директории для моделей, голосов и кэша
сделать так, чтобы run_lexo_mvp.bat мог поднять готовый локальный TTS-контур

Жёсткое требование:
никаких обязательных ручных шагов вне папки LEXO, кроме наличия Python, Flutter и базовой Windows-среды.

12. Сценарий разработки

Порядок, в котором нужно идти:

Этап 0. Контракт и каркас
зафиксировать TTS API
определить DTO для voice, job, tts_segment
решить, как app обращается к engine

Этап 1. TTS data layer
добавить новые таблицы и файловый кэш
подготовить хранение статусов и путей к аудио

Этап 2. TTS segmentation
вынести сегментацию в отдельный модуль
сделать независимый от движка слой разбиения текста

Этап 3. Provider abstraction
создать TtsProvider
сделать пустую или mock-реализацию для быстрого теста
подготовить KokoroProvider

Этап 4. Kokoro integration
подключить реальный локальный запуск Kokoro
получить генерацию одного тестового сегмента
получить список голосов
проверить сохранение wav-файла

Этап 5. Queue and cache
сделать генерацию списка сегментов
добавить пропуск уже готовых аудиофайлов
добавить базовый job status

Этап 6. Reader playback
подключить воспроизведение аудио в Flutter
связать Reader с очередью сегментов
реализовать базовые кнопки управления

Этап 7. Practical validation
прогнать книгу на реальном txt
проверить повторный запуск
проверить смену голоса
проверить, что кэш реально переиспользуется

13. Что считать полностью завершённым этапом

MVP3 считается пройденным, если выполняются все условия:

уже импортированная книга может быть озвучена локально
озвучка работает через отдельный TTS pipeline
текст режется на естественные сегменты
Kokoro реально подключён и генерирует локальное аудио
есть минимум несколько рабочих голосов на выбор
аудио сохраняется внутри LEXO
повторная генерация уже готовых сегментов не запускается
Reader умеет проигрывать сегменты по порядку
архитектура не прибита намертво к одному способу запуска модели
внутри проекта есть setup-сценарий и run-сценарий для TTS-контура

14. Что остаётся следующим этапом после MVP3

Следующим этапом уже можно делать:
словарь слов
сохранение слов
озвучку отдельного слова или фразы
улучшение UX reader
расширение набора голосов
подключение второго TTS backend

Но на этапе MVP3 задача одна:
получить первый действительно рабочий локальный контур книжной озвучки внутри LEXO.
