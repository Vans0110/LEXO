Главное исправление

В твоём тексте смешаны два разных шага:

сборка .ipa
подпись и установка на iPhone

Для твоего сценария самый реалистичный путь такой:

Правильная цепочка

Windows → GitHub Actions на macOS собирает .ipa → ты скачиваешь .ipa → Sideloadly на Windows подписывает и ставит на iPhone через твой Apple ID

То есть не надо пытаться бесплатно настраивать полноценную Apple signing-цепочку прямо внутри GitHub Actions как основной путь. Для бесплатного личного теста это лишняя боль. GitHub удобно использовать именно как удалённую macOS-сборку, а подписывать/ставить уже локально через Sideloadly. GitHub действительно даёт бесплатные стандартные hosted runners для public repositories, включая macOS. Flutter официально требует macOS/Xcode для iOS-сборки. Sideloadly умеет ставить и обновлять IPA на iPhone, а с бесплатным Apple account приложение живёт 7 дней и потом его надо переустанавливать.

Что из твоего текста верно
Верно
GitHub Actions — реально подходит как бесплатная удалённая macOS-сборка, если репозиторий public.
Sideloadly — подходит как установщик/обновлятор на Windows для личного iPhone. Он умеет overwrite поверх старой версии, если тот же bundle ID и тот же Apple ID.
Developer Mode на iPhone нужен для sideloading на iOS 16+.
Основную разработку реально имеет смысл вести на Windows и прогонять большую часть UI/логики через Android/desktop, а iOS проверять отдельными сборками. Это уже не “официальная рекомендация”, а инженерно самый практичный режим при твоих ограничениях.
Где у тебя ошибка

Фраза:

“Один раз создаёшь provisioning profile и сертификат в Apple Developer Portal бесплатно, сохраняешь в GitHub Secrets, и всё подписывается автоматически”

Вот это я бы не брал как базовый план.
Apple официально пишет, что с обычным Apple Account ты можешь тестировать на своих устройствах, но их бесплатный сценарий завязан на Xcode. А GitHub signing workflow в официальной документации GitHub предполагает, что у тебя уже есть сертификат и provisioning profile для CI. Для твоего кейса без Mac и без платного membership это не тот путь, с которого стоит начинать.

Этап 1. Подготовить проект как ios-adaptation

Сейчас у тебя база заточена под Windows. Значит первая задача — не сборка, а отделение Windows-специфики.

Что делаем:

всё Windows-only выносим в отдельные адаптеры
файлы, TTS, локальное хранилище, вызовы внешних сервисов — через интерфейсы
UI перестаём мыслить как desktop-окно и делаем как mobile-first экран
Flutter layout делаем под узкий экран, safe areas, клавиатуру, вертикальный сценарий

То есть не “переписать всё”, а создать ветку:
feature/ios-adaptation

Этап 2. Сделать первый iOS-совместимый интерфейс

Твоя цель на первом проходе не “весь LEXO на айфоне”, а вот это:

один экран
открывается
показывает книгу/текст
есть базовые кнопки
работает навигация
интерфейс не разваливается на iPhone

Это уже даст тебе реальную картину, что именно ломается на iOS.

Этап 3. Настроить удалённую сборку

Тут нужен GitHub Actions workflow на macOS.

Что он делает
checkout repo
ставит Flutter
подтягивает зависимости
делает iOS build
публикует .ipa как artifact

Для старта тебе нужен workflow, который делает именно build artifact, а не App Store/TestFlight release.

Этап 4. Скачивать .ipa и ставить через Sideloadly

На Windows:

подключаешь iPhone
открываешь Sideloadly
выбираешь новый .ipa
вводишь Apple ID
жмёшь Start

Если bundle ID тот же, Sideloadly может перезаписать предыдущую сборку. С бесплатным аккаунтом приложение работает 7 дней, потом его надо снова sideload’ить. Также у бесплатного аккаунта есть лимит на количество sideloaded apps одновременно.

Этап 5. Цикл разработки

Твой цикл будет такой:

Ежедневный режим
80–90% времени — правки на Windows
быстрые проверки — Android emulator / Windows run
когда нужен iPhone check:
commit
push
GitHub Action build
скачать .ipa
Sideloadly install
проверить на телефоне

Да, это не hot reload на iPhone. Но это самый дешёвый путь войти в iOS без Mac.

Честно о скорости
Что будет быстро
push
скачивание артефакта
установка через Sideloadly
Что будет медленно
сама iOS-сборка в GitHub Actions

Поэтому твоя тактика должна быть такой:

мелочь проверять на Android/Desktop
на iPhone гонять пакет правок, а не каждый пиксель

Иначе ты утонешь во времени ожидания.

Android-эмулятор = iPhone?
Нет

Полностью — нет.

Android/Windows тебе дадут:
общую логику
поведение Flutter UI
навигацию
работу состояний
большинство layout-ошибок
Но только iPhone даст:
реальные safe areas
поведение клавиатуры iOS
iOS-специфику плагинов
жесты, системные модалки, поведение шрифтов, скролла, inset’ов

Поэтому Android-эмулятор — не замена iPhone. Но это хороший предфильтр, чтобы не гонять каждую мелочь через облачную iOS-сборку.
