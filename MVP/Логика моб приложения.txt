Правильная схема
1. Одна mobile-разработка

Ты делаешь один Mobile UI

Не:

отдельно iOS UI
отдельно Android UI

А:

один mobile UI
который потом работает и на iPhone, и на Android

Потому что ты сам правильно сказал:

iPhone и Android у тебя будут на 90% одинаковые
2. Одна desktop-разработка

Отдельно у тебя есть Desktop UI
Это твой текущий Windows-формат

3. Платформенные папки — это не “2 отдельные разработки”

Вот тут главное не запутаться.

ios/

Ответ на твой главный вопрос
“Мне делать разработку под iOS/Android или папку mobile?”
Правильно:

делать папку/слой Mobile UI

Потому что:

iOS и Android у тебя почти одинаковы по интерфейсу
различия будут в платформенных вызовах, а не в основном UI
То есть схема такая
Ты делаешь:
Mobile UI
тестируешь его сначала на Android emulator
потом этот же Mobile UI проверяешь на iPhone
Очень важно
Ты не разрабатываешь iOS через Android

Ты разрабатываешь Mobile UI
А Android Emulator используешь как быстрый локальный просмотр mobile-интерфейса

То есть:

Было

“Я делаю iOS и тестирую как Android?”

Стало

“Я делаю общий mobile UI и сначала проверяю его через Android Emulator, потому что на Windows это единственный нормальный mobile preview”

Это разные вещи.

Как это работает по этапам
Этап 1

Ты создаёшь mobile-экран

Например:

mobile home
mobile reader
mobile controls
Этап 2

Запускаешь это на Android Emulator
И смотришь:

влезает ли интерфейс
не разваливается ли layout
удобно ли нажимать
нормальная ли логика навигации
Этап 3

Когда mobile UI уже живой
Ты отправляешь его в iPhone pipeline
И смотришь уже:

safe area
iOS особенности
баги платформы
реальные проблемы iPhone
Чего не надо делать
Не надо
делать 2 mobile UI: один для iPhone, другой для Android
писать отдельно экран под Android и отдельно под iPhone без причины
тащить всю платформенную логику в UI
думать, что ios/ и android/ — это места для двух разных интерфейсов
Что реально может отличаться между iOS и Android

Да, отличия будут, но обычно точечные:

Но это не повод делать 2 разных mobile-приложения

Самая простая схема
LEXO
├── core                  # общая логика
├── ui
│   ├── desktop           # Windows UI
│   └── mobile            # общий UI для iPhone/Android
├── platform
│   ├── windows           # адаптеры Windows
│   ├── ios               # адаптеры iOS
│   └── android           # адаптеры Android
├── ios                   # Flutter iOS runner / native config
└── android               # Flutter Android runner / native config
В одном предложении

Ты не делаешь две параллельные мобильные разработки под iOS и Android — ты делаешь один Mobile UI, тестируешь его сначала на Android Emulator как быстрый mobile-preview, а потом этот же Mobile UI проверяешь на реальном iPhone.
